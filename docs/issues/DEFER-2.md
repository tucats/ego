# DEFER-2 — `deferByteCode` skips receiver capture when `deferThisSize == 0`

**Affected function:** `deferByteCode`  
**File:** `bytecode/defer.go`  
**Risk:** Low — the compiler currently only pushes receivers for deferred method
calls that already have a non-empty receiver stack; a `deferThisSize == 0` context
represents a top-level (non-nested) defer where no method receiver is expected  
**Discovered by:** `Test_deferByteCode_ReceiverCapture_ZeroDeferThisSize_Captures`  
**Status: RESOLVED** (defensive fix; was never an observable bug — see below)

## DEFER-2: Description

The capture block in `deferByteCode` was guarded by `c.deferThisSize > 0`:

```go
if c.deferThisSize > 0 && (c.deferThisSize < len(c.receiverStack)) {
    receivers = c.receiverStack[c.deferThisSize:]  // (after DEFER-1 fix)
    c.receiverStack = c.receiverStack[:c.deferThisSize]
}
```

When `deferThisSize == 0` (i.e. the receiver stack was empty when
`deferStartByteCode` ran), the block was never entered.  If a receiver IS pushed
while evaluating the deferred call's target — a deferred method call made from a
context where no receivers existed yet — that receiver would never be captured
in the `deferStatement` and never trimmed from the live stack.

## DEFER-2: Why it was never an observable bug

An audit (July 2026) established that the guarded capture block is unreachable
in all currently generated bytecode, for two independent reasons:

1. **`defer` is a statement.** It is compiled at a statement boundary, where
   the receiver stack is always empty — receivers are pushed and popped within
   the evaluation of a single call expression, and a `defer` statement can
   never appear part-way through one. So `deferThisSize` (captured by
   `deferStartByteCode` as `len(c.receiverStack)`) is **always 0** for real
   code.

2. **The deferred call is closure-wrapped.** `compileDefer` rewrites
   `defer f()` into `defer func(){ f() }()` (and freezes any receiver chain
   into a temp via `hoistDeferReceiver`, the BUG-43 fix). All receiver setup
   (`SetThis`/`LoadThis`) therefore happens *inside* the closure body, which
   runs later when the defer fires — never on the parent context's live
   `receiverStack` between `DeferStart` and `Defer`. So `len(c.receiverStack)`
   at `deferByteCode` is also always 0.

Both together mean `deferThisSize == len(c.receiverStack) == 0` at
`deferByteCode` in real programs, so neither the DEFER-1 formula nor the
DEFER-2 guard ever affected observable behavior. This was verified end-to-end:
top-level `defer p.Method()`, chained receivers `defer a.b.Method()`, and
receiver-freezing (`x` rebound after `defer x.Method()`) all bind the correct
receiver and do not corrupt subsequent method calls.

## DEFER-2: Fix

The redundant, asymmetric `deferThisSize > 0` conjunct was removed so the guard
is simply `c.deferThisSize < len(c.receiverStack)`. This is a no-op for all
current bytecode (the condition is `0 < 0` → false), but is now
correct-by-construction should a future defer form ever leave a receiver
pending with `deferThisSize == 0`: everything on the stack then belongs to the
deferred call and is captured (and trimmed) rather than silently dropped.

`Test_deferByteCode_ReceiverCapture_ZeroDeferThisSize_Captures` (renamed from
`..._NoCapture` and inverted) pins the corrected behavior.
